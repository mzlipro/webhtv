# 夸克网盘登录态迁移实验记录

记录日期：2026-06-09

## 结论

本次实验成功：将一台手机上接口 Jar 内置网盘登录能力产生的夸克登录态文件备份到电脑，再恢复到另一台手机的同包名 App 私有目录后，夸克登录状态可以恢复。

这次实验没有发现登录态主要保存在 WebView Cookie 数据库，也不是普通 `files` 目录下的 cookie 文本。成功恢复所用的重点文件为：

```text
shared_prefs/spUtils.xml
shared_prefs/myDrive_useState.xml
files/quark_fid
cache/.ftyfnw3ynpan2ka3
```

其中最关键的登录凭据位于：

```text
shared_prefs/spUtils.xml
```

该文件包含 `quark_ck`。实验记录中不保存、不打印 `quark_ck` 原值。

## 环境

- App 包名：`com.fongmi.android.tv`
- App 版本：`5.5.2`
- versionCode：`552`
- 私有目录：`/data/user/0/com.fongmi.android.tv`
- 分支：`investigate/login-cookie-storage-20260609`
- 源手机：已登录夸克网盘
- 目标手机：先安装 release 包时无法 `run-as`，手动安装 debug 包后可恢复

目标手机安装 release/non-debuggable 包时，ADB 返回：

```text
run-as: package not debuggable: com.fongmi.android.tv
```

目标手机没有 `su`，普通 ADB shell 也不能访问 `/data/user/0/com.fongmi.android.tv`。因此实验恢复依赖 debug 包的 `run-as` 能力。这里的 debug 包不是 root 权限，只是允许 ADB 以 App 自身 uid 访问该 App 私有目录。

## 登录态定位过程

登录前后做了 App 私有目录快照：

```text
/private/tmp/webhtv-cookie-before.tsv
/private/tmp/webhtv-cookie-after.tsv
```

登录后初始新增文件为：

```text
shared_prefs/myDrive_useState.xml
shared_prefs/spUtils.xml
```

字段检查结果：

```text
shared_prefs/spUtils.xml
  string name="quark_ck"
  quark_ck_length=168

shared_prefs/myDrive_useState.xml
  boolean name="quark_useState"
  quark_useState=true
```

`quark_ck` 看起来是 base64-like 字符串，解码后不是可读 JSON，更像接口 Jar 自己编码或加密后的登录态缓存。本记录不包含它的原始值。

WebView Cookie 数据库检查：

```text
app_webview/Default/Cookies
```

结果：未发现 Quark/UC/网盘相关域名 cookie。该库中只有之前影视站点相关 cookie，因此这次接口 Jar 登录态不是靠 WebView CookieManager 持久化。

`files` 目录检查：

```text
files/TV/.qktime
files/TV/.deleteqktime
files/quark_fid
```

`find files -type f` 会包含隐藏文件。登录初期 `files` 下没有发生关键变化；后续出现的 `files/quark_fid` 内容为 `{}`，只有 2 字节，不是 cookie 本体。

另外发现一个可疑缓存文件：

```text
cache/.ftyfnw3ynpan2ka3
```

该文件是二进制/低可读比例缓存，未发现可读的 Quark/cookie/token 标记。本次成功恢复时把它一起带上了，但尚未单独验证它是否必需。

## 备份结果

电脑本地备份目录：

```text
/private/tmp/webhtv-quark-login-backup-20260609
```

重点登录态普通目录树：

```text
/private/tmp/webhtv-quark-login-backup-20260609/login-focus-files
```

包含 4 个文件：

```text
cache/.ftyfnw3ynpan2ka3
files/quark_fid
shared_prefs/myDrive_useState.xml
shared_prefs/spUtils.xml
```

全量变动普通目录树：

```text
/private/tmp/webhtv-quark-login-backup-20260609/all-changed-files
```

包含 29 个文件。该目录是按登录前快照到当前快照的新增、mtime 或大小变化提取出来的备份。实验实际恢复成功只使用了重点登录态 4 个文件，没有恢复全量 29 个文件。

同时保留了未压缩 tar，主要用于 ADB 恢复时一次性传输：

```text
/private/tmp/webhtv-quark-login-backup-20260609/quark-login-focus.tar
/private/tmp/webhtv-quark-login-backup-20260609/quark-login-all-changed.tar
```

注意：用户明确表示重点不是压缩包。后续开发和说明应优先使用普通目录树概念，tar 只是传输实现细节。

## 恢复过程

目标手机手动安装 debug 包后，确认 `run-as` 可用：

```bash
adb exec-out run-as com.fongmi.android.tv pwd
```

返回：

```text
/data/user/0/com.fongmi.android.tv
```

恢复前检查目标手机 4 个重点文件：

```text
MISSING ./shared_prefs/spUtils.xml
MISSING ./shared_prefs/myDrive_useState.xml
MISSING ./files/quark_fid
MISSING ./cache/.ftyfnw3ynpan2ka3
```

推送恢复包：

```bash
adb push /private/tmp/webhtv-quark-login-backup-20260609/quark-login-focus.tar /data/local/tmp/webhtv-quark-login-focus.tar
```

用 App 自身身份解包到私有目录：

```bash
adb exec-out run-as com.fongmi.android.tv sh -c 'cd /data/user/0/com.fongmi.android.tv && tar -xf /data/local/tmp/webhtv-quark-login-focus.tar'
```

恢复后验证：

```text
shared_prefs/spUtils.xml              276 bytes, quark_ck 存在，长度 168
shared_prefs/myDrive_useState.xml     121 bytes, quark_useState=true
files/quark_fid                       2 bytes
cache/.ftyfnw3ynpan2ka3               102440 bytes
```

随后执行：

```bash
adb shell am force-stop com.fongmi.android.tv
```

用户重新打开 App 后确认：夸克登录态恢复成功。

## 注意事项

1. 不要在日志、文档、终端输出中打印 `quark_ck` 原值。它应被视为登录凭据。
2. release/non-debuggable 包不能通过普通 ADB `run-as` 写入私有目录。
3. 无 root 设备无法直接访问 `/data/user/0/com.fongmi.android.tv`。
4. 如果产品化该能力，不能依赖用户安装 debug 包。需要 App 内部提供导入/导出入口。
5. 恢复前建议先停止 App，避免 SharedPreferences 内存态覆盖磁盘文件；恢复后也应 `force-stop` 或重启 App 让配置重新加载。
6. 本次成功恢复用了 4 个重点文件，但还没有拆分验证每个文件是否必需。当前判断：
   - `spUtils.xml` 的 `quark_ck` 是核心凭据。
   - `myDrive_useState.xml` 是登录启用状态。
   - `files/quark_fid` 当前内容为 `{}`，可能不是关键。
   - `cache/.ftyfnw3ynpan2ka3` 是否必需未知，本次为了稳妥一起迁移。

## 对后续开发的启示

当前一键同步里的“Jar/脚本保存数据”默认关注外部存储路径，例如 `TV`、`TVBox`、`TVData`。这次实验说明，接口 Jar 的网盘登录态可能写在 App 私有目录的 `shared_prefs` 和 `cache` 中，不一定写在外部存储。

如果要做正式功能，建议设计为独立的“网盘登录态备份/恢复”能力：

- 明确提示这是敏感数据，默认关闭，用户手动触发。
- 导出时只收集明确白名单文件或白名单 key。
- 不在日志中输出凭据值。
- 可考虑对导出内容做本地加密或至少加风险提示。
- 恢复时先停止相关运行任务，再写入文件，最后重启或提示用户重启 App。
- 需要兼容 release 包，因此应由 App 内部以自身权限完成导入/导出，而不是依赖 ADB `run-as`。

当前最小白名单可从以下路径开始：

```text
shared_prefs/spUtils.xml
shared_prefs/myDrive_useState.xml
files/quark_fid
cache/.ftyfnw3ynpan2ka3
```

后续可以继续做拆分实验，确认是否只需要 `spUtils.xml` 和 `myDrive_useState.xml`。
